정의

VPN의 정의

VPN(Virtual Private Network, 가상 사설망)은 인터넷처럼 누구나 접근 가능한 공용(Public) 네트워크 위에, 인증과 암호화를 통해 특정 사용자·조직만 접근할 수 있는 논리적인 사설(Private) 통신 구간을 만드는 기술입니다.

물리적으로는 다른 사람들과 같은 인터넷 회선을 공유하지만, 터널링(Tunneling)과 암호화를 통해 마치 자신들만의 전용선을 쓰는 것과 같은 효과를 얻습니다. 이름 그대로 "가상(Virtual)"의 "사설망(Private Network)"인 셈입니다.

역할

VPN의 근본 역할은 ① 터널링(공용망 위에 논리적 통로를 만드는 것), ② 암호화(그 통로 안의 내용을 제3자가 볼 수 없게 하는 것), ③ 인증(통로의 양 끝이 신뢰할 수 있는 상대인지 확인하는 것), 이 세 가지를 결합해 공용망을 사설망처럼 안전하게 쓸 수 있도록 만드는 것입니다. 이후 다루는 IPsec의 모든 세부 설정(Phase1/2)은 이 세 가지 역할을 구체적으로 어떻게 구현할지에 대한 파라미터입니다.

01

VPN의 종류

세부 구조로 들어가기 전에, VPN을 나누는 두 가지 큰 기준과 각 유형이 실제로 어떤 역할을 하는지 정리합니다.

① 프로토콜 기준 — IPsec VPN vs SSL VPN

유형역할
IPsec VPN 네트워크 계층(L3)에서 IP 패킷 자체를 암호화·캡슐화하여, 두 네트워크(또는 장비와 네트워크)를 마치 하나의 사설망처럼 연결하는 역할을 합니다. 한 번 터널이 맺어지면 그 안의 모든 트래픽(어떤 애플리케이션이든)이 자동으로 암호화되어 오가므로, "통로 자체"를 만드는 인프라 역할에 가깝습니다. 그래서 본사-지사처럼 양쪽 네트워크를 상시로 통합해야 하는 상황에 적합합니다.
SSL VPN TLS(HTTPS와 동일한 암호화 계층)를 이용해, 개인 사용자가 웹 브라우저나 클라이언트로 특정 리소스에 접근할 수 있게 해주는 역할을 합니다. 네트워크 전체를 잇는 것이 아니라 "이 사람이 이 리소스에 접근해도 되는지"를 개별적으로 통제하는 접근 관문 역할이 강합니다. 그래서 사용자별로 접근 범위를 세밀하게 나누고 싶은 재택근무·외부 접속 시나리오에 적합합니다.
구분IPsec VPNSSL VPN
동작 계층네트워크 계층(L3)응용/전송 계층 (TLS 기반)
접속 방법FortiClient 등 전용 클라이언트 또는 상호운용 장비 필요웹 브라우저만으로 포털 접속 가능(Tunnel Mode는 클라이언트 필요)
접근 범위네트워크 전체(서브넷 단위) 접근에 유리특정 애플리케이션/리소스 단위 접근에 유리
성능하드웨어 가속(NPU) 지원으로 대체로 빠름상대적으로 오버헤드가 있을 수 있음
대표 용도본사-지사 상시 연결, 사이트 간 통합외부 웹포털을 통한 간편 원격 접속

② 목적 기준 — Site-to-Site VPN vs Remote-Access VPN

유형역할
Site-to-Site VPN 장비(FortiGate)와 장비 사이에 터널을 맺어, 서로 다른 위치의 네트워크 두 개를 하나의 네트워크처럼 이어주는 역할입니다. 일반 사용자는 자신이 VPN을 타고 있다는 사실조차 인지하지 못하며(투명하게 동작), 관리자가 한 번 구성해두면 계속 유지되는 "상시 배관" 역할을 합니다.
Remote-Access VPN 개인 단말이 필요할 때마다 사내망에 접속할 수 있도록 일시적인 개인용 출입구를 열어주는 역할입니다. 사용자가 직접 FortiClient를 실행하거나 로그인해야 연결되며, 접속을 끊으면 그 순간 통로도 사라지는 "온디맨드 접근" 역할에 가깝습니다.
구분Site-to-Site VPNRemote-Access VPN
연결 대상네트워크 ↔ 네트워크개인 단말 ↔ 네트워크
목적지사·데이터센터·클라우드 간 상시 통합재택근무·출장자의 임시/개인 접속
사용자 인지 여부일반 사용자는 VPN 존재를 인지하지 못함(투명)사용자가 직접 접속을 시작(FortiClient 실행 등)

역할

이 네 가지 분류는 뒤에 나올 모든 세부 설정(Network, Authentication, Phase1/2 등)을 "어떤 목적으로, 어떤 프로토콜로 구성할 것인가"를 먼저 정하는 상위 결정입니다. 이 결정에 따라 이후 항목의 선택값(Remote Gateway, Mode Config 등)이 달라집니다.

FortiGate가 이 두 목적(Site-to-Site / Remote-Access)을 실제로 어떤 연결 구조(1:1, 1:N, Dialup 등)로 구현하는지는 3. 연결 구조 유형에서 이어서 다룹니다.
02

IPsec Tunnel을 적용하는 이유

인터넷과 같은 공개망 위에 암호화된 가상의 전용선을 만들어, 값비싼 전용회선 없이도 안전한 사설 네트워크 연결을 구축하기 위함입니다.

목적설명
기밀성ESP 암호화로 데이터 탈취되어도 내용 확인 불가
무결성해시(HMAC)로 전송 중 변조 여부 검증
인증PSK / 인증서로 상대가 진짜인지 확인

역할

기밀성·무결성·인증이라는 3대 가치를 통해, 신뢰할 수 없는 공용망을 신뢰할 수 있는 사설망처럼 바꿔주는 것이 IPsec Tunnel의 근본 역할입니다. 이후에 나오는 모든 세부 항목(Phase1/2)은 이 3가지 가치를 구체적으로 어떻게 구현할지를 정하는 파라미터일 뿐입니다.

03

연결 구조 유형

FortiGate IPsec Wizard에서 제공하는 4가지 템플릿의 핵심 차이입니다. 이는 앞서 다룬 Site-to-Site / Remote-Access 목적을 실제로 어떤 구조로 구현하는지를 보여줍니다.

Remote GatewayStatic IPDialup UserDialup User수동 선택
먼저 접속양쪽 가능스포크 → 허브사용자 → 허브설계에 따라 다름
Mode Config보통 없음있음있음선택 가능

스포크(Spoke)란

Hub-and-Spoke 구조에서 허브(중앙)에 연결되는 개별 지사·지점입니다. 스포크끼리는 직접 연결되지 않고 허브를 경유하며, Spoke-to-Spoke 통신이 필요하면 허브에 라우팅과 방화벽 정책을 추가해야 합니다.

Dialup User란

상대 IP를 미리 지정하지 않고, PSK·Peer ID·인증서 등 인증만으로 접속을 허용하는 Remote Gateway 방식입니다. 상대가 먼저 접속해야 하며, 여러 상대가 동시에 접속할 수 있습니다. IP가 유동적인 원격 사용자, 다수 소규모 지점, FortiClient 접속에 사용됩니다.

Point-to-Point vs Point-to-Multipoint

위 4가지 구조는 결국 아래 두 가지 연결 형태 중 하나에 속합니다.

구분연결 형태대표 구성
Point-to-Point1개의 중심 ↔ 1개의 상대Site to Site (Static IP)
Point-to-Multipoint1개의 중심(허브) ↔ 여러 개의 상대(스포크)Dialup / Hub-and-Spoke

Point-to-Multipoint 환경에서는 스포크마다 터널 인터페이스 IP를 수동으로 배정하기 어렵기 때문에, Exchange Interface IP 같은 자동화 기능이 함께 사용됩니다.

역할

이 섹션은 "터널을 몇 개의 상대와, 어떤 방향으로 맺을 것인가"를 결정하는 설계 단계입니다. 여기서 정해진 구조(1:1 / 1:N)에 따라 뒤에 나올 Network 섹션의 Remote Gateway 값, Peer Option의 Accept Types 값이 자동으로 갈립니다.

04-1

Network - Advanced

Add Route

Phase 2 협상이 성공했을 때, 상대(Remote Address) 대역을 라우팅 테이블에 자동으로 등록할지 여부를 결정합니다. Exchange Interface IP와 함께 쓰는 BGP 환경에서는 라우팅을 BGP가 전담하도록 보통 Disable로 둡니다.

Auto Discovery Sender

허브(Hub) FortiGate에서 활성화합니다. ADVPN에서 스포크 A와 스포크 B가 서로의 IP/대역 정보를 모를 때, 허브가 중개자 역할을 하며 두 스포크에게 상대 정보를 알려주는(Discovery) 역할을 합니다.

Auto Discovery Receiver

스포크(Spoke) FortiGate에서 활성화합니다. 허브로부터 받은 Discovery 정보를 바탕으로, 다른 스포크와 직접 IPsec 터널(Shortcut Tunnel)을 동적으로 생성합니다.

Exchange Interface IP

Dialup 방식의 Point-to-Multipoint 연결에서, 허브와 스포크가 IKE 협상 중 터널 인터페이스(오버레이) IP를 자동으로 서로 교환하도록 합니다. 수동 IP 설정 없이도 BGP 등 동적 라우팅 이웃 관계를 자동 구성할 수 있습니다.

Device Creation (net-device)

Dialup 세션마다 개별 인터페이스(디바이스)를 만들지(Enable, 라우팅 기반) 아니면 하나로 공유할지(Disable, Selector 기반) 결정합니다. 아래 Aggregate Member 구성 시 반드시 Disable이어야 합니다.

Aggregate Member

여러 WAN 회선의 IPsec 터널을 하나의 논리 인터페이스로 묶어 부하분산·이중화를 구현합니다. net-device Disable이 전제조건이며, Dialup 터널은 멤버로 추가할 수 없습니다.

기타 항목역할
Backup GatewayStatic 상대 게이트웨이 장애 시 전환할 백업 IP 목록

역할

04번의 기본 Network 항목이 "무엇을 켤지"를 정한다면, 04-1 Advanced는 "여러 회선·여러 상대를 다루는 고급 토폴로지(ADVPN, 회선 이중화, 동적 라우팅 연동)"를 구성하는 항목들을 모아둔 곳입니다. Add Route·Auto Discovery·Exchange Interface IP는 라우팅 자동화 흐름으로, Device Creation·Aggregate Member는 회선 이중화 흐름으로 서로 연결되어 있습니다.

Diffie-Hellman Group 번호의 의미

DH Group 번호는 키 교환에 사용하는 수학적 그룹의 종류와 크기를 나타내는 식별자입니다. 번호가 클수록 무조건 안전한 것이 아니라, 어떤 방식(MODP/타원곡선)의 그룹인지와 키 길이가 함께 바뀝니다.

Group방식키 길이비고
1MODP768bit매우 취약, 사용 금지
2MODP1024bit구형, 비권장
5MODP1536bit구형, 비권장
14MODP2048bit현재 표준적인 최소 권장 수준
15MODP3072bit더 높은 보안, 연산 부하↑
19ECP(타원곡선)256bitMODP보다 짧은 키로 동등 이상의 강도, 연산 효율 좋음
20ECP(타원곡선)384bit고보안 환경 권장
21ECP(타원곡선)521bit최고 수준 보안
MODP(Modular Exponential)는 소수 기반의 전통적인 방식으로 키 길이가 길어질수록 연산량이 급격히 늘어나고, ECP(Elliptic Curve, 타원곡선)는 훨씬 짧은 키로 동일하거나 더 높은 보안 강도를 낼 수 있어 최신 환경에서 선호됩니다. 실무에서는 Group 14(무난한 기본) 또는 Group 19 이상(고보안·고효율)을 주로 사용하며, Group 1/2/5는 더 이상 사용하지 않는 것이 좋습니다.

역할

Phase 1 Proposal은 "IKE SA(관리 채널) 자체를 얼마나 강한 암호화로 보호할 것인가"를 정하는 곳입니다. 이 채널이 뚫리면 그 안에서 오가는 Phase2 협상 내용까지 노출될 수 있으므로, 사실상 전체 터널 보안의 기초 체력을 결정하는 항목입니다.

Phase 1 vs Phase 2 Key Lifetime의 차이

구분Phase 1 Key LifetimePhase 2 Key Lifetime
보호 대상IKE SA (관리 채널)IPsec SA (실제 데이터 트래픽)
판단 기준시간(초)만 가능시간(Seconds) / 트래픽량(KBytes) / 둘 다(Both) 선택 가능
기본값86400초 (24시간)43200초 (12시간)
재협상(Rekey) 빈도상대적으로 낮음 (관리 채널이라 자주 바꿀 필요 적음)상대적으로 높음 (실제 트래픽이 많이 오갈수록 한 키에 노출되는 데이터량이 늘어나므로)

Phase 1은 "협상을 위한 관리 채널"이라 하루 한 번 정도 갱신해도 충분하지만, Phase 2는 실제 데이터가 그 키로 암호화되어 흐르는 채널이므로 트래픽량이 많은 고속 회선에서는 시간이 아직 안 지났어도 일정 용량(KBytes)에 도달하면 먼저 재협상하도록 만들 수 있습니다. 이는 하나의 키로 암호화되는 데이터량을 제한해, 키가 유출되었을 때의 피해 범위와 암호 분석 위험을 줄이기 위한 설계입니다.

역할

New Phase2 Advanced는 "실제 데이터를 얼마나 강하게 암호화하고, 그 키를 얼마나 자주·안전하게 갱신할 것인가"를 정하는 곳입니다. 앞의 Encryption/Authentication/PFS는 보호 강도를, Auto-negotiate/Autokey Keep Alive/Key Lifetime은 SA의 생명주기 관리를 담당한다는 두 축으로 이해하면 됩니다.
